Locales for Linux

One of my current occupations is pushing for interest in the localization (often abbreviated l10n) of the Linux operating system to as many languages as possible; for that I created some supplementary locales support in addition to those distributed with GNU libc. They are available here in rpm package format, handy for those of you that use that format. Otherwise, for languages not supproted by default by GNU libc, the sources of locale/charset definitions are available. THe compiled locales can be created by running: localedef -i xx_YY -f ZZZZZ -u mnemonic.ds xx
. With xx being the iso language code, YY the iso country code, and ZZZZ the charset to be used.

You can read a good explanation about localization, its meaning, its implications, and some vocabulary, from Mozilla i18n & l10n Guidelines page and the comprehensive Concepts of C/UNIX Internationalization page of Dave Johnson.

And you can visit this very interesting and complete site on charsets encodings: http://czyborra.com/charsets/

How to use them?

That is indeed quite easy.
First you need to install the package corresponding to the language(s) you want support for, of course.
After that you need to instruct Linux which language(s) to use, there are several variables that deal with the locales, first each locale category has its own: LC_CTYPE, LC_TIME, LC_COLLATE, LC_NUMERIC, and LC_MONETARY, and the LC_ALL that overwrites and sets all the previous ones. So you can choose different locales for each category, but that is seldom done. There are also two other environnement variables that deal with locales: LANG and LANGUAGE they are quite similar and work like LC_ALL in that they define the default locales for all categories, but contrary to LC_ALL, they don't overwrite explicitely defined LC_* variables. So, for facility, generally only LANG is used to define the locale. LANGUAGE is somewhat a GNU extension to LANG, not all programs recognize it (but very few don't, and they are each time lesser), but the nice thing is that it allows to specify multiple locales, in order of preference. That is very handy, specially for people using a language which has few translations, but that know also other languages; or to people using minoritary languages.
So my advice is to use LANG if you only need one locale, and both LANG and LANGUAGE if you want several ones.

The format

The format of a locale name is ll[_CC[.EEEE]][@dddd] where ll is the two letters iso code for languages (ISO 639), in lowercase; CC is the two letter iso code for countries (ISO 3166), in uppercase; EEEEE is the charset encoding, in uppercase (there isn't really any standardization on the format; so it is generally omitted); and ddddd is an eventual dialectal variant, when the country or the charset encoding is not enough to distinguish two locales of the same language (exemples of that are no (Bokmål Norwegian) and no@nynorsk (Nynorsk Norwegian); or my *@euro that use the euro for the monetary symbol, or es@tradicional which uses the traditional sorting order of spanish, etc...)

Finally...

So, from the above, you need to edit your $HOME/.profile (or whatever file is sourced at login time, be it $HOME/.Xclients, $HOME/.xinitrc or others) and put:

export LANG=xx
export LANGUAGE=xx:yy:zz:aa_BB
With xx being your preferred language, yy, zz, aa_BB, etc being other possible locales you may want.

That's all !
Now all programs will correctly display the letters of your language (no more filtring of 8th bit by less or tin...), and (provided they had been transalted) man pages, programs interfaces and menus, etc, will show in your language.
Enjoy !

  • Arabic

    The arabic locale package only have basic data, to tell the weekdays and month names, the alphabetical order; but it doesn't deal with the right-to-left displaying problem; nor with the handling of initial/medial/final/separate form of the letters. For that you will need an arabic aware terminal. I found one some time ago; but it was very old, and the C code not much portable; I have been unable to compile it on Linux. If you know about arabic enabled software let me know.

    It seems something is finally on the way ! Look at this http://www.linux4arab.com/ page (it is in arabic though, so it is a little hard to use if you haven't yet right-to-left support, as it is probably the case...
    Myself I can't read arabic, so if someone could tell me what it provides (eg is there only a news site, or does it provide arabized versions of some programs, toolkits, etc.). Thanks.

    And here another interesting link: http://www.planux.com/linux/. it is written in english about topics related to Linux in arabic (news, book reviews, LUGS, software, etc)

    Link to Arabic enabled Mozilla project. From there you can go to several interesting places about computer handling of RTL languages (Arabic, Hebrew)

    The Fribidi library is a functions library that can do bidirectional conversion of text strings.

    I added support for iso-8859-6-8, iso-8859-6-16, ASMO 449+, cp1256, xaterm-fontspecific (extended encoding used by the fonts coming with xaterm) and the mulearabic-{O,1,2} font encodings to the xtt font server. My additions should be still considered beta, as several points are still open:

  • the name of the fonts under X11
  • the unicode values of some chars

    There is an arabic on Linux mailing-list, you can join it by writting an email to arabic-linux-request@marko.net, it is definitively the place to be to discuss about the work to do to add arabic support to Linux (there is still a lot to do, all the most used graphical toolkits and either linux console or ncurses (or both?) must resolve the RTL/LTR, at least). More info at http://www.marko.net/arabic/

  • Japanese

    As the current version of glibc (2.0) doesn't have full support for multibyte charsets I have needed to cheat a little; rewritting the locale definition to use the char definitions of iso-8859-1 and telling localedef to use iso-8859-1. As a result the sorting (LC_COLLATE) is wrong for non latin letters. The LC_CTYPE also isn't totally correct; but I make it so the functions like isprint() etc are able to correctly display japanese strings.

    That means that, in practice, the japanese locales fullfill its role; putting LANG=ja you have date in japanese, and the programs are able to display japanese text.

  • Thai

    The Thai locale doesn't correctly work for alphabetical sorting in glibc 2.0; that is a bug on GNU libc 2.0; the collating-element directive only works when the elements are 7bit ascii only; if the elements are 8bit (like in Thai language) that doesn't work.
    The good news is that is fixed in GNU libc 2.1 and not only that, but glibc 2.1 includes a full Thai locale (but it isn't compiled by default on any Linux distribution I know of)

    There is an open source project to add Thai support on software, you can reach it at http://software.thai.net/. The locale definition I use comes from there.

    The Thai Reference page has useful links to several documents about handling of Thai language in computers.

    Check also the Thai Linux Workgroup (note however that most of the pages are in Thai, you need Thai fonts to browse them).

    The ZzzThai's Linux Resources a must page for Linux and Thai. You can find there the Thai-HOWTO and the TE (Thai Extensions) package.

    The Mozilla Thai Enabling project.

    I added support for tis620.2533-1 encoding to the xtt font server.

    As the current version of glibc (2.0) doesn't have full support for multibyte charsets I have needed to cheat a little; rewritting the locale definition to use the char definitions of iso-8859-1 and telling localedef to use iso-8859-1. As a result the sorting (LC_COLLATE) is wrong for non latin letters. The LC_CTYPE also isn't totally correct; but I make it so the functions like isprint() etc are able to correctly display chinese strings.

    The package contains support for locales zh_CN, which uses the GuoBiao encoding used in Mainland China; and zh_TW, which uses the Big5 encoding in use in Taiwan.
